Like you're 5

Hooks are like seats on a bus that's always loaded in the exact same order every trip. If you skip a seat one day and not the next, everyone's seat number gets confused.

  • Only call Hooks at the top level — never inside loops, conditions, or nested functions.
  • Only call Hooks from React functions — function components or other custom Hooks, never plain JS functions.
  • React tracks Hooks by the order they're called in, render after render. Breaking the order desyncs which state belongs to which useState call.
  • The eslint-plugin-react-hooks ESLint rule catches violations automatically — always keep it enabled.
Interview tip: Say it plainly: "Hooks rely on call order, not names, to match state across renders — that's why conditional hook calls break everything."
Like you're 5

Passing a plain number is like shouting a guess at someone far away who might not have heard your last shout yet. Passing a function is like handing them a note that says "whatever the number is when you read this, add one."

  • const [state, setState] = useState(initialValue) returns the current value and a setter function; calling the setter schedules a re-render with the new value.
  • Direct form setCount(count + 1) uses the value of count captured at render time — if called multiple times in the same event, it uses the stale value each time.
  • Functional update form setCount(c => c + 1) always receives the latest pending state, so multiple updates in a row correctly stack.
  • Use the functional form whenever the new state depends on the previous state, especially inside loops, timers, or rapid event handlers.
function handleClick() {
  setCount(count + 1); // stale — all three see the same `count`
  setCount(count + 1);
  setCount(count + 1); // net effect: +1
}

function handleClickFixed() {
  setCount(c => c + 1); // always fresh
  setCount(c => c + 1);
  setCount(c => c + 1); // net effect: +3
}
Interview tip: This exact triple-click example is a favorite live-coding interview question — know it cold.
Like you're 5

It's a shopping list you hand the effect: "only go back to the store if one of these specific items changed."

  • The second argument to useEffect(fn, deps) tells React when to re-run the effect.
  • Omitted array: runs after every single render.
  • Empty array []: runs once, after the first render only.
  • Array with values [a, b]: re-runs whenever any listed value changes between renders (compared with Object.is).
  • Every value from component scope used inside the effect (props, state, functions) should be listed — the ESLint exhaustive-deps rule flags missing ones, since they cause stale-closure bugs.
useEffect(() => {
  document.title = `${count} clicks`;
}, [count]); // re-runs only when count changes
Interview tip: If asked "what happens if you lie to the dependency array," describe stale closures — the effect keeps referencing an old value forever.
Like you're 5

Before the stagehands set up the next scene, they first tidy up whatever was left over from the current one.

  • The function returned from inside useEffect is the cleanup function.
  • It runs before the effect runs again (when a dependency changes) and when the component unmounts.
  • This is where you cancel subscriptions, clear timers/intervals, remove event listeners, or abort in-flight requests — anything the effect set up that would otherwise leak.
  • In React 18 Strict Mode (development only), React deliberately runs mount → cleanup → mount again once, to help you notice missing or broken cleanup logic.
useEffect(() => {
  const id = setInterval(() => console.log('tick'), 1000);
  return () => clearInterval(id); // cleanup
}, []);
Interview tip: Forgetting cleanup is the #1 cause of memory leaks and "why is this running twice/why do I have duplicate listeners" bugs — always mention it unprompted.
Like you're 5

useMemo remembers a finished cake so you don't have to bake it again. useCallback remembers the recipe card itself so you're not handed a brand-new piece of paper every time, even if the recipe didn't change.

  • useMemo(fn, deps) memoizes the return value of an expensive calculation — recomputed only when a dependency changes.
  • useCallback(fn, deps) memoizes the function reference itself — useful so a child wrapped in React.memo doesn't see a "new" function prop every render.
  • useCallback(fn, deps) is functionally equivalent to useMemo(() => fn, deps).
  • Both are performance optimizations, not correctness tools — overusing them adds complexity and its own small overhead, so reserve them for genuinely expensive computations or to stabilize props passed to memoized children.
const sorted = useMemo(() => expensiveSort(list), [list]); // memoize a VALUE

const handleClick = useCallback(() => {
  doSomething(id);
}, [id]); // memoize a FUNCTION reference
Interview tip: Clean one-liner: "useMemo caches a value, useCallback caches a function — both exist to keep referential equality stable across renders."
Like you're 5

A ref is like a sticky note on your fridge — you can write on it and read it anytime, but writing on it doesn't ring a doorbell to announce it to anyone.

  • useRef(initialValue) returns a mutable object { current: initialValue } that persists across renders without causing one.
  • Two main uses: (1) holding a reference to a DOM node (<div ref={myRef}>) to imperatively read/measure/focus it, and (2) holding any mutable value that shouldn't trigger a re-render when it changes (timers IDs, previous values, render counts).
  • Mutating ref.current does not re-render the component — this is the key difference from useState.
  • Because it doesn't trigger renders, reading ref.current during render (rather than in an effect/handler) can show stale or inconsistent values — refs are meant for side-channel, not render-driving, data.
function TextInput() {
  const inputRef = useRef(null);
  useEffect(() => { inputRef.current.focus(); }, []);
  return <input ref={inputRef} />;
}
Interview tip: Key contrast to nail: useState triggers a re-render and is for anything the UI displays; useRef doesn't and is for anything the UI doesn't need to react to.
Like you're 5

Instead of five friends passing a note hand-to-hand down a line, useContext is like a loudspeaker — anyone standing anywhere in the room can just tune in and hear the message directly.

  • createContext(defaultValue) creates a Context object; a <Context.Provider value={...}> makes a value available to every descendant, no matter how deeply nested.
  • Any descendant calls useContext(MyContext) to read that value directly — skipping every intermediate component that would otherwise need to forward it as a prop.
  • This removes the need for components that don't care about the data to still declare and pass it along.
  • Trade-off: every component that calls useContext re-renders whenever the Provider's value changes, so very frequently-changing values (e.g. mouse position) are a poor fit for Context; split contexts by concern to limit blast radius.
const UserContext = createContext(null);

function App() {
  return <UserContext.Provider value={{ name: 'Ana' }}><Profile /></UserContext.Provider>;
}

function Profile() {
  const user = useContext(UserContext); // reads directly, no props needed
  return <p>{user.name}</p>;
}
Interview tip: Mention the re-render trade-off unprompted — it shows you understand Context isn't a free performance win, just a wiring convenience.
Like you're 5

useState is one light switch — fine for one bulb. useReducer is a whole control panel with labeled buttons, better when many related switches need to move together in predictable ways.

  • const [state, dispatch] = useReducer(reducer, initialState) manages state via a pure reducer function (state, action) => newState, similar to Redux's core idea.
  • You update state by calling dispatch({ type: 'increment', payload: 5 }) instead of setting values directly.
  • Preferred over useState when: state is a complex object with several sub-values that change together, the next state depends on the previous one in non-trivial ways, or the update logic itself is complex enough to want testing in isolation from the component.
  • Keeps update logic centralized and predictable, and makes it easy to trace "what actions led to this state" — genuinely useful for debugging complex forms or workflows.
function reducer(state, action) {
  switch (action.type) {
    case 'increment': return { count: state.count + action.payload };
    case 'reset': return { count: 0 };
    default: throw new Error('Unknown action');
  }
}

const [state, dispatch] = useReducer(reducer, { count: 0 });
dispatch({ type: 'increment', payload: 5 });
Interview tip: Good line: "useReducer is useState's bigger sibling for when the shape of the update logic itself deserves its own testable function."
Like you're 5

useEffect is like checking your outfit in the mirror after you've already walked out the door. useLayoutEffect checks and fixes it before anyone even sees you leave the house.

  • useLayoutEffect has the identical API to useEffect, but it fires synchronously after DOM mutations, before the browser paints the screen.
  • useEffect fires asynchronously after paint — the user may briefly see the pre-effect UI first.
  • Use useLayoutEffect when you need to measure or mutate the DOM before the user sees it — e.g. measuring an element's size/position and adjusting layout to avoid a visible flicker (tooltip positioning is the classic example).
  • Default to useEffect for everything else (data fetching, subscriptions, logging) since blocking paint unnecessarily hurts perceived performance.
useLayoutEffect(() => {
  const { height } = ref.current.getBoundingClientRect();
  setTooltipTop(height + 8); // must happen before paint to avoid flicker
}, []);
Interview tip: If you don't have a strong reason to block paint, use useEffect — useLayoutEffect should be the exception, not the default.

Normally when you hand someone a walkie-talkie (a ref), they can press every single button on it. useImperativeHandle lets you tape over most buttons and expose only the two or three you actually want them to use.

  • useImperativeHandle(ref, () => ({...})) customizes the value exposed when a parent attaches a ref to a child component (used together with forwardRef).
  • Instead of exposing the raw underlying DOM node, the child controls exactly which methods/values the parent is allowed to call — e.g. focus(), scrollToTop(), or a custom validation trigger.
  • This keeps the child's internal implementation details private while still allowing a small, deliberate imperative API for cases React's declarative props can't easily express (focus management, animations, imperative form resets).
  • Used sparingly — most component communication should stay via props, not imperative refs.
const FancyInput = forwardRef((props, ref) => {
  const inputRef = useRef();
  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current.focus(),
  }));
  return <input ref={inputRef} />;
});

// Parent: fancyInputRef.current.focus()  — only focus() is exposed, nothing else
Interview tip: Mention it's an escape hatch for imperative needs (focus, scroll, animation triggers) — not a general communication pattern.
Like you're 5

If you find yourself giving the same set of instructions to three different friends, you write the instructions down once on a card and just hand each friend the same card.

  • A custom Hook is simply a JavaScript function whose name starts with use and that calls other Hooks inside it — it's how React lets you extract and reuse stateful logic.
  • Write one whenever the same stateful logic (not just JSX) is duplicated across multiple components — e.g. fetching data, tracking window size, debouncing a value, subscribing to an event.
  • Custom Hooks don't share state between components automatically — each call gets its own independent state, exactly like calling useState directly in each component would.
  • They compose naturally: a custom Hook can call other custom Hooks, letting you build up rich, testable, reusable behavior without any component nesting/wrapper overhead (unlike HOCs).
function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);
  useEffect(() => {
    const onResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', onResize);
    return () => window.removeEventListener('resize', onResize);
  }, []);
  return width;
}

function Component() {
  const width = useWindowWidth(); // reused anywhere
  return <p>{width}px wide</p>;
}
Interview tip: Clarify the common misconception: custom Hooks reuse LOGIC, not STATE — two components calling the same custom Hook get two separate, independent copies of that state.
Like you're 5

If two people in a play both introduce themselves as "Sam," the audience gets confused about who's who. useId hands every actor a unique nametag so the accessibility tools always know exactly who's speaking.

  • useId() generates a unique, stable ID string for the lifetime of a component instance — useful for wiring up accessibility attributes like <label htmlFor> / id pairs, or aria-describedby.
  • Solves the problem of hardcoding IDs (which collide if the component renders more than once on the page) or generating random IDs manually (which mismatch between server and client during hydration).
  • It's specifically designed to be consistent across server and client rendering in SSR apps, avoiding hydration-mismatch warnings that ad-hoc Math.random() IDs would cause.
  • Not meant as a key for lists, and not meant to generate visual content — purely for accessibility wiring and unique DOM attribute IDs.
function Field({ label }) {
  const id = useId();
  return (
    <>
      <label htmlFor={id}>{label}</label>
      <input id={id} />
    </>
  );
}
Interview tip: Mention the SSR hydration-safety angle specifically — that's the detail that separates a memorized answer from real understanding.
Like you're 5

Imagine typing fast in a search box while a picture is being repainted behind the scenes. useDeferredValue lets your typing stay instant and snappy while quietly letting the slow repaint catch up a moment later.

  • const deferredValue = useDeferredValue(value) gives you a version of a value that lags behind during urgent updates, letting React prioritize more urgent rendering (like keystrokes) first.
  • It's part of React's Concurrent features — React can interrupt rendering the deferred (lower-priority) value if something more urgent comes in.
  • Common use: type into a search input immediately (urgent), while a large, expensive results list re-renders using the deferred value slightly behind, keeping the input itself buttery smooth.
  • Different from debouncing: there's no fixed delay timer — React decides dynamically based on actual rendering pressure, and it can render immediately if the main thread is free.
function Search({ query }) {
  const deferredQuery = useDeferredValue(query);
  const results = useMemo(() => expensiveSearch(deferredQuery), [deferredQuery]);
  return <ResultsList results={results} />; // input stays snappy even if this is slow
}
Interview tip: Contrast with debouncing explicitly: useDeferredValue is priority-based and adaptive, not a fixed timer — that nuance impresses interviewers.
Like you're 5

It's like putting a "please wait, still loading" sticky note on a big slow task, while letting you keep clicking other buttons in the meantime without everything freezing.

  • const [isPending, startTransition] = useTransition() lets you mark a state update as a non-urgent transition, so React can keep the UI responsive to more urgent updates while it happens.
  • isPending is a boolean you can use to show a loading indicator while the transition is in progress.
  • Best for: switching tabs/views that trigger a heavy re-render, filtering a large list, or any UI update that isn't a direct, immediate response to typing/clicking but still needs to feel non-blocking.
  • Unlike useDeferredValue (defers a value), useTransition defers an update you trigger — both are part of React's Concurrent Rendering toolkit and are often used together.
function TabSwitcher() {
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();

  function selectTab(next) {
    startTransition(() => setTab(next)); // marked as non-urgent
  }

  return <div>{isPending ? 'Loading…' : <Content tab={tab} />}</div>;
}
Interview tip: Clean distinction to state out loud: useDeferredValue defers a VALUE you already have; useTransition defers an UPDATE you trigger.

It's like having x-ray glasses that let you peek inside a toy to see exactly which gears are turning, instead of just guessing from the outside.

  • The Components tab of React DevTools lists every mounted component; selecting one shows its live hooks list in order (State, Effect, Memo, etc.) with current values.
  • You can edit state values live from the panel to test different scenarios without changing code and reloading.
  • The Profiler tab records a session and shows a flamegraph of which components rendered, how long each took, and why they re-rendered (via the "Render reasons" option) — invaluable for chasing unnecessary re-renders.
  • Hooks appear unnamed by default ("State", "Effect") but you can label them using the useDebugValue hook inside a custom Hook for clearer custom-hook debugging.
function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(true);
  useDebugValue(isOnline ? 'Online' : 'Offline'); // shows nicely in DevTools
  return isOnline;
}
Interview tip: Mentioning useDebugValue for custom hooks is a small detail that signals real hands-on DevTools experience.
Like you're 5

Imagine calling roll for the same 5 kids every single day in the exact same order. If one day you skip kid #3 depending on the weather, everyone after them shifts up one seat and gets the wrong name called.

  • No — Hooks must always be called in the exact same order on every render of a given component.
  • React doesn't track Hooks by name; it tracks them by a simple internal linked list / array index tied to call order. A conditional or looped Hook call shifts every subsequent Hook's slot, corrupting state association.
  • If you need conditional behavior, put the condition inside the Hook instead of around it — e.g. always call useEffect, but check the condition inside its callback.
  • For genuinely variable numbers of stateful items (like a dynamic list of fields), model the state as a single array/object in one useState/useReducer call rather than calling useState in a loop.
// ❌ Breaks call order
if (isLoggedIn) {
  useEffect(() => { trackLogin(); }, []);
}

// ✅ Always call the hook, branch inside
useEffect(() => {
  if (isLoggedIn) trackLogin();
}, [isLoggedIn]);
Interview tip: Explain the WHY (call-order tracking), not just the rule — interviewers specifically probe whether you understand the mechanism, not just memorized "don't do it."
Like you're 5

You can't hand a walkie-talkie itself back as a sandwich — useEffect expects either nothing back or a cleanup function, not a promise. So you make a little helper inside to do the async part.

  • useEffect's callback cannot itself be async, because an async function always returns a Promise, and React expects either undefined or a cleanup function as the return value — not a Promise.
  • The fix: define an async function inside the effect and call it immediately (or extract it and call it), rather than marking the effect callback itself async.
  • Always guard against a race condition: if the component unmounts or dependencies change before the request finishes, ignore the stale result using a boolean flag or an AbortController.
  • This pattern is also where cleanup shines — cancelling the in-flight request in the returned cleanup function.
useEffect(() => {
  let cancelled = false;

  async function fetchData() {
    const res = await fetch('/api/user');
    const data = await res.json();
    if (!cancelled) setUser(data); // guard against stale/late responses
  }

  fetchData();
  return () => { cancelled = true; }; // cleanup
}, []);
Interview tip: Always mention the race-condition guard unprompted — forgetting it is the classic bug where an old, slow request overwrites a newer, faster one's result.
Like you're 5

useMemo is you manually double-checking your own math homework every time. The React Compiler is having a very smart tutor who automatically checks and skips redoing any math that hasn't changed — without you asking.

  • useMemo/useCallback are manual memoization — the developer decides what to wrap and lists dependencies by hand, with real risk of bugs from incomplete dependency arrays.
  • The React Compiler (formerly "React Forget") is a build-time tool that automatically analyzes component code and inserts memoization for values/functions/components, without any manual useMemo/useCallback/React.memo calls needed.
  • It relies on components following the Rules of React (pure rendering, no unauthorized side effects during render) to safely reason about what can be memoized.
  • Goal: get most of the performance benefit of manual memoization for free, reducing both boilerplate and the class of bugs caused by stale/incorrect dependency arrays.
Interview tip: Frame it as "a shift from developer-authored memoization to compiler-inferred memoization" — shows awareness of where React is heading, not just where it's been.
Like you're 5

Instead of wrapping a gift box inside another gift box inside another gift box, you just write the instructions once on a card that anyone can read and use themselves.

  • Write a custom Hook that encapsulates the shared stateful logic, and call it directly inside each component that needs it.
  • Unlike an HOC, this adds no extra component layer to the tree — no "wrapper hell," no prop-name collisions, no obscured component displayed names in DevTools.
  • For sharing rendered UI/markup (not just logic), use composition — pass components via children or named slot props, rather than a HOC injecting extra props.
  • Custom Hooks have effectively replaced HOCs and Render Props for logic-sharing in modern React codebases.
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue(v => !v), []);
  return [value, toggle];
}

function Modal() {
  const [isOpen, toggleOpen] = useToggle(false); // reused directly, no wrapper component
}
Interview tip: If pressed on "when would you still use an HOC today," a fair answer is: rarely — mostly in older codebases, or for cross-cutting concerns tightly bound to routing libraries.
Like you're 5

It's like a form that remembers what happened the last time you tried submitting it — including whether it's still working on your request right now — without you needing to track any of that yourself.

  • const [state, formAction, isPending] = useActionState(actionFn, initialState) ties a Server Action (or any async action function) directly to form state.
  • state holds the latest result returned by the action (e.g. validation errors or a success message); formAction is passed to <form action={formAction}>; isPending tracks whether the action is still running.
  • It removes the need to manually wire up useState for the result, a loading flag, and an onSubmit handler that calls preventDefault and awaits the request — React manages that plumbing.
  • Designed to pair with React 19's Server Actions and progressive-enhancement-friendly forms, but works with any async function, server or client.
async function submitAction(prevState, formData) {
  const result = await saveName(formData.get('name'));
  return result.error ? { error: result.error } : { success: true };
}

function NameForm() {
  const [state, formAction, isPending] = useActionState(submitAction, {});
  return (
    <form action={formAction}>
      <input name="name" />
      <button disabled={isPending}>{isPending ? 'Saving…' : 'Save'}</button>
      {state.error && <p>{state.error}</p>}
    </form>
  );
}
Interview tip: Mention it replaced the earlier-named useFormState — naming history like this signals you're actively tracking React 19 changes, not just reciting docs.
