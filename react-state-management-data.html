Like you're 5

Your own toothbrush stays in your own bathroom (local state). But the TV remote in the living room is shared by the whole family (global state) — you wouldn't want five separate remotes that don't agree with each other.

  • Local state (useState/useReducer inside a component) is right when only that component and maybe its direct children need the data — a toggle, an input's current text, a modal's open/closed flag.
  • Global state (Context, Redux, Zustand, etc.) is right when many unrelated parts of the app need to read or update the same data — the logged-in user, theme, shopping cart, feature flags.
  • Rule of thumb: start local. Lift state up only as far as the nearest common ancestor that actually needs it; reach for a global store only when lifting would mean threading data through many unrelated layers.
  • Overusing global state for things that are really local hurts performance (more components subscribe to changes) and makes the app harder to reason about.
Interview tip: A strong answer explicitly says "default to local, escalate only when you have a real cross-cutting need" — it shows judgment, not just tool knowledge.
Like you're 5

Context is like a shared family group chat — great for the whole family to see a message, but a bit clunky if you want fine controls like muting one topic. Redux/Zustand is a proper app built just for organizing lots of conversations efficiently.

  • Context API is built into React for passing a value down the tree without prop drilling — it's a dependency-injection mechanism, not a full state-management solution.
  • Context has no built-in way to prevent re-renders of every consumer when the value changes, and no dev tools, middleware, or built-in async handling.
  • Redux/Zustand are dedicated state libraries: they offer fine-grained subscriptions (only re-render components that use the specific slice that changed), middleware for logging/async, time-travel debugging (Redux DevTools), and a clear separation of state logic from components.
  • General guidance: Context is great for rarely-changing, app-wide values (theme, locale, auth user); a real state library is better for frequently-changing, complex, or large-scale shared state.
Interview tip: Interviewers specifically like hearing "Context re-renders every consumer on any value change unless you split it carefully" — that's the crux of the trade-off.

Old Redux was like building furniture with only raw wood and no instructions — technically possible, very tedious. Redux Toolkit hands you the pre-cut pieces and a clear manual.

  • Redux Toolkit is the official, recommended way to write Redux — it drastically cuts the boilerplate that plain Redux required.
  • createSlice() generates action creators and a reducer together from one object, and uses Immer under the hood so you can write "mutating" code that's actually converted to immutable updates.
  • configureStore() sets up the store with good defaults already wired in — Redux DevTools support, the thunk middleware, and helpful development checks (like catching accidental state mutations).
  • RTK Query, bundled with RTK, adds a full data-fetching and caching layer, often removing the need for separate manual thunks for API calls entirely.
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => { state.value += 1; }, // looks mutating, isn't
  },
});

const store = configureStore({ reducer: { counter: counterSlice.reducer } });
Interview tip: If asked "would you use plain Redux today," the honest answer is no — RTK is the standard, and plain Redux is considered legacy.

Redux is like filling out several forms in triplicate to change one setting. Zustand is like just flipping the switch directly — no forms, no ceremony.

  • Zustand stores are created with a single create() call — no actions, reducers, or providers required; you define state and update functions together in one place.
  • Components subscribe with a simple hook const value = useStore(state => state.value), and by default only re-render when the selected slice changes — fine-grained reactivity with almost no setup.
  • No <Provider> wrapping is required — the store lives outside React entirely and can even be read/updated from non-React code.
  • Much smaller bundle size and far less boilerplate than Redux, at the cost of some of Redux's ecosystem maturity (time-travel debugging, middleware ecosystem, strict conventions for very large teams).
const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
}));

function Counter() {
  const count = useStore((s) => s.count);
  const increment = useStore((s) => s.increment);
  return <button onClick={increment}>{count}</button>;
}
Interview tip: Good framing: Zustand trades some of Redux's structure and tooling maturity for dramatically less boilerplate — the right call for small-to-mid apps and fast-moving teams.

Instead of one giant shared whiteboard where everyone has to scan the whole thing to find their note, atomic state gives everyone their own small sticky note that only the people who need it look at.

  • Atomic state management breaks global state into many small, independent units called atoms, each holding one piece of state.
  • Components subscribe only to the specific atoms they use, so updating one atom re-renders only its subscribers — not the whole tree, and not even other unrelated global state.
  • Atoms can be derived from other atoms (computed/combined values) similarly to spreadsheet formulas, and Jotai in particular integrates tightly with React Suspense for async atoms.
  • Compared to one big Redux/Zustand store, atomic state tends to reduce unnecessary re-renders by default and scales naturally as features are added independently, without needing to carefully structure one large state tree upfront.
const countAtom = atom(0);
const doubledAtom = atom((get) => get(countAtom) * 2); // derived atom

function Counter() {
  const [count, setCount] = useAtom(countAtom);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
Interview tip: Compare it directly to a spreadsheet: atoms are like cells, derived atoms are like formula cells that auto-recalculate — a very relatable analogy in interviews too.

Data on your own device is like your sock drawer — you fully control it. Data from a server is like borrowing a book from the library — it can change, expire, or need to be returned; React Query is the librarian that manages all of that borrowing for you.

  • Server state (data that actually lives on a server) behaves differently from client/UI state: it can be stale, shared across components, needs re-fetching, and can fail — using plain useState + useEffect for this reinvents a lot of tricky logic badly.
  • React Query handles caching, background re-fetching, deduplication of identical requests, retries, pagination, and stale-data invalidation out of the box.
  • It gives you isLoading, isError, data states automatically per query key, and re-fetches intelligently on window refocus or reconnect — all things developers previously wrote by hand, often incorrectly (missing race-condition guards, no caching, etc).
  • The mental model shift: don't put server data in your global client-state store at all — let React Query own it, and keep Redux/Zustand/Context for genuinely client-only UI state.
function UserProfile({ id }) {
  const { data, isLoading, error } = useQuery({
    queryKey: ['user', id],
    queryFn: () => fetch(`/api/users/${id}`).then(r => r.json()),
  });
  if (isLoading) return <Spinner />;
  if (error) return <ErrorMessage />;
  return <p>{data.name}</p>;
}
Interview tip: The line "server state and client state are fundamentally different problems" is the single strongest interview soundbite for this question.
Like you're 5

Instead of downloading the entire library's books at once, you ask the librarian for just "page 2" of the catalog, and React Query remembers what you already fetched so it doesn't ask twice.

  • Include the page number in the query key so each page is cached independently: ['todos', page].
  • Set placeholderData: keepPreviousData (React Query v5) so the UI keeps showing the previous page's data (instead of a loading spinner) while the next page loads — a much smoother UX.
  • For infinite scroll instead of numbered pages, use useInfiniteQuery, which manages fetching and merging successive pages and exposes a fetchNextPage() function tied to a cursor or offset from the API response.
  • Each page's data is cached under its own key, so navigating back to a previously-visited page is instant (served from cache) unless it's gone stale.
const { data, isPlaceholderData } = useQuery({
  queryKey: ['todos', page],
  queryFn: () => fetchTodos(page),
  placeholderData: keepPreviousData,
});
Interview tip: Mention keepPreviousData by name — it's the detail that shows hands-on usage rather than surface-level familiarity.

When you press a "like" button, you don't want to stare at a loading spinner — you want the heart to turn red instantly, as if you already know it worked, and only quietly fix it if it turns out it didn't.

  • An optimistic update immediately updates the UI to reflect the expected result of an action, before the server confirms it succeeded.
  • This makes the app feel instant, since the user doesn't wait for a network round-trip to see feedback.
  • If the request fails, you must roll back to the previous state and typically show an error — React Query's onMutate/onError/onSettled callbacks on useMutation are built exactly for this: snapshot the previous cache, apply the optimistic change, and restore the snapshot on failure.
  • Only appropriate for actions that are very likely to succeed and where a brief inconsistency is an acceptable trade-off — not for something like a payment charge.
const mutation = useMutation({
  mutationFn: likePost,
  onMutate: async (postId) => {
    const prev = queryClient.getQueryData(['posts']);
    queryClient.setQueryData(['posts'], (old) => markLiked(old, postId)); // optimistic
    return { prev };
  },
  onError: (err, postId, context) => {
    queryClient.setQueryData(['posts'], context.prev); // rollback
  },
});
Interview tip: Always mention the rollback path — describing optimistic updates without the failure case is an incomplete answer in most interviews.

A cache is like leftovers in the fridge — fast to grab again without recooking. But sometimes the leftovers go bad, and you need to know exactly when to throw them out and cook fresh.

  • Caching stores previously-fetched data keyed by something unique (a query key, a URL) so repeat requests can be served instantly without hitting the network again.
  • Invalidation is marking cached data as stale/outdated so it gets refetched — typically after a mutation that changes the underlying data (e.g. creating a new todo should invalidate the todos list).
  • In React Query, queryClient.invalidateQueries({ queryKey: ['todos'] }) after a successful mutation is the standard pattern — it marks matching queries stale and triggers a background refetch for any that are currently active.
  • Two knobs matter: staleTime (how long data is considered fresh before a refetch is even attempted) and gcTime/cacheTime (how long unused data stays in memory before being garbage collected).
const mutation = useMutation({
  mutationFn: createTodo,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['todos'] }); // marks stale, triggers refetch
  },
});
Interview tip: Know the difference between staleTime and gcTime specifically — it's a very common follow-up question after this one.

It's like handing someone yesterday's newspaper right away so they have something to read immediately, while a fresh one is quietly being printed in the background to swap in the moment it's ready.

  • Stale-While-Revalidate (SWR) serves cached (possibly outdated) data immediately for a fast perceived response, while simultaneously firing a background request to fetch fresh data.
  • Once the fresh data arrives, the UI silently updates to reflect it — the user never sees a blocking loading state for data they've seen before.
  • This is the default behavior of both the SWR library (by Vercel) and React Query's caching model — it's a general caching strategy, not tied to one library.
  • Trade-off: the user might briefly see slightly stale data, which is usually an acceptable price for a much snappier feel, especially for data that doesn't need to be instantly accurate.
Interview tip: If asked to name the HTTP header equivalent: the Cache-Control: stale-while-revalidate directive implements the same idea at the HTTP caching layer, not just in JS libraries.
Like you're 5

You don't want to keep asking the same question over and over while waiting for an answer, and you definitely don't want to still be listening for an answer after you've already left the room.

  • Prefer a dedicated data-fetching library (React Query, SWR) over hand-rolled useEffect + fetch — it correctly handles caching, deduplication, and cleanup that's easy to get wrong manually.
  • If fetching manually: call the request inside useEffect(() => {...}, []), guard against setting state after unmount (a cancellation flag or AbortController), and handle loading/error states explicitly.
  • Never fetch directly in the component body (outside an effect) — that would refetch on every single render, an infinite loop risk if the fetch itself triggers a state update.
  • Consider whether the data is actually needed before the component even mounts — frameworks like Next.js let you fetch during routing/server rendering, avoiding an extra client-side request and loading flash entirely.
useEffect(() => {
  const controller = new AbortController();
  fetch('/api/data', { signal: controller.signal })
    .then(r => r.json())
    .then(setData)
    .catch(err => { if (err.name !== 'AbortError') setError(err); });
  return () => controller.abort();
}, []);
Interview tip: If the interviewer's stack uses Next.js/Remix, mention fetching at the routing/server layer as the more modern-preferred alternative to client useEffect fetches.
Like you're 5

If you order a slow pizza and then change your mind and order a fast burger, you don't want the pizza showing up last and being served to you instead of the burger you actually wanted now.

  • A race condition happens when an earlier, slower request resolves after a later, faster request — overwriting the UI with stale data even though it was technically requested first.
  • Fix with a cancellation flag: set a local boolean to true in the effect's cleanup, and check it before applying the response.
  • Better: use an AbortController, so the outdated request is actually cancelled at the network level, not just ignored.
  • Dedicated libraries (React Query, SWR) solve this automatically — each query is keyed, and only the latest request for a given key is allowed to update state, which is one of the strongest reasons to prefer them over hand-rolled fetching.
useEffect(() => {
  let ignore = false;
  fetchResults(query).then(data => {
    if (!ignore) setResults(data); // ignore stale/late responses
  });
  return () => { ignore = true; };
}, [query]);
Interview tip: Walk through a concrete scenario (fast typing in search) rather than defining it abstractly — interviewers want to see you recognize it as a real bug pattern.
Like you're 5

It's like a cancel button on a food delivery app — if you change your mind, you press cancel and the kitchen actually stops cooking your old order instead of it showing up anyway later.

  • AbortController is a native browser/Node API for cancelling in-flight async operations, most commonly fetch requests.
  • const controller = new AbortController() gives you controller.signal (passed to fetch(url, { signal })) and controller.abort() to cancel it.
  • In React, create the controller inside a useEffect, pass its signal to the fetch, and call controller.abort() in the cleanup function — so navigating away, unmounting, or re-running the effect (dependency change) properly cancels the stale request instead of leaving it dangling.
  • Aborted fetches throw a DOMException named 'AbortError' — always check for and ignore that specific error rather than treating it as a real failure.
useEffect(() => {
  const controller = new AbortController();
  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then(r => r.json())
    .then(setResults)
    .catch(err => { if (err.name !== 'AbortError') console.error(err); });
  return () => controller.abort();
}, [query]);
Interview tip: Explicitly mention catching and ignoring AbortError — forgetting that turns a clean cancellation into a scary-looking console error.

Debouncing is waiting until someone finishes an entire sentence before you respond, instead of interrupting after every single word. Throttling is answering at most once every few seconds no matter how fast they keep talking.

  • Debouncing delays acting until a pause in activity — e.g. only fire the search request 300ms after the user stops typing, cancelling any pending timer on each new keystroke.
  • Throttling guarantees an action fires at most once per fixed time interval, regardless of how continuously the trigger fires — useful for scroll or resize handlers.
  • In React, implement debouncing with a useEffect that sets a setTimeout on every keystroke and clears the previous one in cleanup, or with useDeferredValue for a more "let React decide" approach.
  • For search inputs specifically: debounce the API call itself, but keep the input's displayed value fully controlled and instant — never debounce what the user sees themselves typing.
function useDebouncedValue(value, delay) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id); // cancels the previous pending update
  }, [value, delay]);
  return debounced;
}
Interview tip: Say clearly: debounce = wait for a pause; throttle = cap the rate. Mixing these two up in an interview is a common tell of shallow understanding.

Imperative is giving turn-by-turn driving directions ("turn left, then right, then..."). Declarative is just showing the destination on a map and letting the GPS figure out the turns itself.

  • Imperative code describes how to do something, step by step — "find this DOM node, then set its text, then add this class."
  • Declarative code describes what the result should look like for a given state, and leaves the "how" to the underlying system — React's JSX describing UI as a function of state is the textbook example.
  • React itself is declarative at the component level, but still has imperative escape hatches (refs, useImperativeHandle, direct DOM APIs) for the rare cases declarative props/state genuinely can't express (focus, measurements, animations, third-party DOM libraries).
  • The declarative model is what makes React's re-render + diff approach possible and predictable — you never have to reason about every intermediate DOM mutation, only the final desired output.
Interview tip: Good soundbite: "React is declarative by default, imperative by exception — refs exist specifically for the exceptions."

If every single light switch in a huge stadium rings the same alarm bell the instant you flip it, the alarm never stops. Better to have each section only alert its own small area.

  • The core problem: a fully-controlled form with state in one parent component re-renders the entire form on every keystroke in any field, which becomes slow with hundreds/thousands of fields.
  • Use a form library like React Hook Form, which keeps most field state outside React state (using uncontrolled refs internally) and only re-renders on actual validation/submit events — not per keystroke.
  • Isolate state per field rather than one big shared object — wrap each input in its own small component so a keystroke only re-renders that one input, not siblings.
  • For very large/dynamic forms (thousands of rows), combine this with virtualization (react-window) so only the visible rows are even mounted in the DOM at all.
Interview tip: Naming React Hook Form specifically and explaining WHY it's fast (mostly uncontrolled internally) is the strongest possible answer here.
Like you're 5

Manually tracking every field with useState is like writing down every word someone says as they say it. React Hook Form just glances at the whole page once submitted, which is much less tiring.

  • React Hook Form is a form library that minimizes re-renders by keeping field values largely uncontrolled internally (via refs), rather than driving every keystroke through React state.
  • register('fieldName') wires an input up to the form's internal tracking without needing a useState + onChange pair per field.
  • Built-in validation (required, min/max, pattern, custom validators) and integration with schema libraries like Zod or Yup via resolvers, so validation logic lives in one declarative schema instead of scattered handlers.
  • Performance win over manually managed controlled forms: typing in one field doesn't re-render the whole form or sibling fields, since most state changes bypass React's render cycle until submission or explicit validation.
const { register, handleSubmit, formState: { errors } } = useForm();

function onSubmit(data) { console.log(data); }

<form onSubmit={handleSubmit(onSubmit)}>
  <input {...register('email', { required: true })} />
  {errors.email && <span>Required</span>}
</form>
Interview tip: Mention the mostly-uncontrolled-internally detail specifically — it's the actual mechanism behind the performance claim, not just a marketing line.
Like you're 5

Instead of writing a separate list of rules for every single field by hand each time, you write one clear rulebook once, and hand it to both the form and the server so everyone follows the same rules.

  • Zod and Yup are schema-validation libraries: you define the shape and rules of your data once (types, required fields, min/max, custom refinements), and reuse that schema across the client form and the server/API request handler.
  • With React Hook Form, a resolver (zodResolver/yupResolver) plugs the schema in directly, so validation errors populate formState.errors automatically without hand-written validation functions.
  • Zod additionally gives you static TypeScript types inferred directly from the schema (z.infer<typeof schema>), so your form's data type and its validation rules can never drift out of sync.
  • Sharing the same schema on client and server prevents the common bug where client-side validation and server-side validation quietly diverge over time.
const schema = z.object({
  email: z.string().email(),
  age: z.number().min(18),
});

const { register, handleSubmit, formState: { errors } } = useForm({
  resolver: zodResolver(schema),
});
Interview tip: Mention type inference (z.infer) if the interviewer's stack is TypeScript-heavy — it's Zod's standout advantage over Yup.

Normally you'd write a whole separate delivery service (an API route) just to send a letter. Server Actions let you just write "send this letter" directly in your form, and the framework quietly handles delivering it to the server.

  • Server Actions (introduced in React 19 / popularized by Next.js) are async functions marked with 'use server' that run only on the server, but can be called directly from client components as if they were local functions.
  • They can be passed straight to a <form action={myServerAction}>, letting forms submit and mutate data without manually writing a separate API route and a client-side fetch call to it.
  • Because they run server-side, they can safely access databases, secrets, and server-only APIs directly, while the framework handles serializing the call and the response across the network boundary.
  • They pair naturally with useActionState for form status/errors, and with cache revalidation functions (e.g. revalidatePath) to refresh stale data immediately after a mutation.
// actions.js
'use server';
export async function createTodo(formData) {
  await db.todos.create({ text: formData.get('text') });
  revalidatePath('/todos');
}

// component
<form action={createTodo}>
  <input name="text" />
  <button>Add</button>
</form>
Interview tip: Note this collapses the traditional client-fetch + API-route + database round trip into one direct, type-safe function call — that framing shows you understand the architectural shift, not just the syntax.

It's like writing a note and taping it to the fridge so it's still there tomorrow — but you should always double-check the note is still readable before trusting it, in case it got smudged or someone else's fridge doesn't have a note at all.

  • Read/write inside a useEffect (not during render) so it doesn't run during server-side rendering, where localStorage doesn't exist and would throw.
  • Always wrap reads and writes in try/catch — storage can be full, disabled (private browsing in some browsers), or contain corrupted/invalid JSON from an older app version.
  • Validate/parse defensively: JSON.parse can throw, and the shape of old stored data might not match your current app's expectations after an update — consider versioning your stored schema.
  • For SSR frameworks specifically, initializing state from localStorage directly during the first render causes a hydration mismatch (server has no localStorage) — initialize with a default value first, then sync from storage inside an effect after mount.
function useLocalStorage(key, initial) {
  const [value, setValue] = useState(initial);

  useEffect(() => {
    try {
      const stored = localStorage.getItem(key);
      if (stored) setValue(JSON.parse(stored));
    } catch { /* ignore corrupted data */ }
  }, [key]);

  useEffect(() => {
    try { localStorage.setItem(key, JSON.stringify(value)); } catch { /* storage full/disabled */ }
  }, [key, value]);

  return [value, setValue];
}
Interview tip: Bring up the SSR hydration-mismatch risk unprompted — it's the detail that separates someone who's shipped this in a Next.js app from someone who's only tried it in a plain SPA.
