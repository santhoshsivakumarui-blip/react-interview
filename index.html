Like you're 5

Imagine building with LEGO instead of carving a toy out of a single block of wood. Vanilla JS makes you carve and re-carve the whole toy every time something changes. React gives you LEGO pieces (components) you snap together, and it only rebuilds the exact piece that changed.

  • React is a JavaScript library (not a framework) for building user interfaces, created by Meta.
  • It lets you describe the UI declaratively: you say what the UI should look like for a given state, and React figures out how to update the actual screen.
  • Vanilla JS is imperative: you manually find DOM elements (document.querySelector) and mutate them step-by-step every time data changes.
  • React introduces the component model, JSX, and the Virtual DOM to make UI updates predictable and reusable instead of a tangle of manual DOM edits.
// Vanilla JS — you do all the work
const btn = document.querySelector('#btn');
let count = 0;
btn.addEventListener('click', () => {
  count++;
  document.querySelector('#count').textContent = count;
});

// React — you just describe the UI for any given count
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
Interview tip: Interviewers want to hear "declarative vs imperative" and "library vs framework" — React only handles the view layer, unlike Angular which is a full framework.
Like you're 5

Before painting your whole room, you'd sketch it on paper first, compare the sketch to how the room looks now, and only repaint the walls that actually changed color. The Virtual DOM is React's sketch paper.

  • The Virtual DOM (VDOM) is a lightweight JavaScript object tree that mirrors the real DOM structure.
  • When state changes, React builds a new Virtual DOM tree describing the next UI.
  • Reconciliation is the diffing algorithm that compares the new tree against the previous one to find the minimal set of real DOM changes needed.
  • React uses heuristics (the "Fiber" architecture) like comparing elements by type and position, and uses key props to track list items, to make this diff close to O(n) instead of a costly O(n³) full tree comparison.
  • Only the actual differences are applied to the real DOM via the renderer (react-dom), which is why React avoids expensive full re-paints.
Interview tip: Mention "Fiber" by name — it's the reconciler engine since React 16 that made rendering interruptible and enabled features like Concurrent Mode.

A component is the cookie cutter. An element is one cookie stamped out by that cutter — a plain description of what should appear, not the cookie itself yet.

  • A React element is a plain, immutable JavaScript object describing a piece of UI — created by React.createElement(type, props, children) or JSX, which compiles to that call.
  • A component is a function (or class) that returns elements. It's the reusable blueprint/template.
  • You call a component (React calls it internally when it appears in JSX) to produce elements; elements themselves are just data, not something you can call methods on.
  • Because elements are plain objects, React can cheaply create, compare, and throw them away every render — that's part of what makes the Virtual DOM fast.
// This JSX...
const element = <Greeting name="Ana" />;

// ...compiles roughly to this element object:
const element2 = React.createElement(Greeting, { name: 'Ana' });

// Greeting is the COMPONENT (the function)
function Greeting({ name }) {
  return <h1>Hello, {name}</h1>; // this returned value is also an ELEMENT
}
Interview tip: A crisp one-liner interviewers love: "An element is a description of a component instance; a component is the function that produces elements."
Like you're 5

Props are like the instructions printed on a birthday card someone hands you — you can read them, but you're not allowed to scribble over what the sender wrote.

  • Props (short for "properties") are the inputs passed from a parent component to a child, similar to function arguments.
  • Props are read-only / immutable from the receiving component's perspective — a component must never modify its own props.
  • Data flows one way: parent → child. If a child needs to change something, the parent passes down a callback function as a prop, and the child calls that function instead of mutating props directly.
  • This one-way flow (unidirectional data flow) is what keeps React apps predictable and easy to debug.
function Child({ label, onIncrement }) {
  // label = 5;  ❌ NEVER mutate props
  return <button onClick={onIncrement}>{label}</button>;
}

function Parent() {
  const [count, setCount] = useState(0);
  return <Child label={count} onIncrement={() => setCount(c => c + 1)} />;
}
Interview tip: If asked "can you make props mutable?" — technically JS objects are mutable, but doing so breaks React's rendering guarantees and is considered undefined behavior. Always say no.
Like you're 5

It's like a bouncer at a door: if your name's on the list, you get shown one card; if not, a different card (or no card at all).

  • if/else statements before the return, when logic is complex — return different JSX from early returns.
  • Ternary operator condition ? A : B for inline either/or rendering.
  • Logical AND condition && to render something only when true (careful with falsy numbers like 0 rendering as "0").
  • Switch statements or lookup objects when there are many possible states/variants.
  • Returning null from a component renders nothing at all.
function Status({ isLoggedIn, isLoading }) {
  if (isLoading) return <Spinner />;

  return (
    <div>
      {isLoggedIn ? <Dashboard /> : <LoginForm />}
      {isLoggedIn && <LogoutButton />}
    </div>
  );
}
Interview tip: Watch out for the classic bug: {count && } renders a literal "0" on screen when count is 0. Use {count > 0 && } instead.
Like you're 5

Keys are like name tags at a party. Without them, if two guests swap seats, React thinks two brand-new people walked in instead of recognizing the same two people just moved.

  • key is a special prop that helps React's reconciliation identify which items in a list have changed, been added, or removed between renders.
  • Keys must be stable, unique among siblings, and predictable — ideally a database ID, not the array index.
  • Using the array index as a key is risky when the list can be reordered, filtered, or items inserted/removed — React can mismatch component state to the wrong item, causing bugs like inputs showing the wrong value.
  • Without keys (or with bad keys), React falls back to less efficient tree-diffing and may unnecessarily re-mount components, losing internal state and hurting performance.
// ❌ Risky if the list can reorder
todos.map((todo, i) => <TodoItem key={i} {...todo} />)

// ✅ Stable identity
todos.map(todo => <TodoItem key={todo.id} {...todo} />)
Interview tip: Say explicitly: "index as key is only safe when the list is static — never reordered, filtered, or has items inserted in the middle."
Like you're 5

A class component is like an old-school kitchen with dozens of buttons and this.knobs everywhere. A function component is a modern kitchen where you just say what you want and Hooks quietly do the wiring.

  • Class components extend React.Component, use this.state, this.setState(), and lifecycle methods (componentDidMount, etc.).
  • Function components are plain JS functions that return JSX, and use Hooks (useState, useEffect, etc.) for state and side effects instead of lifecycle methods and this.
  • Function components avoid the confusing behavior of JS's this binding entirely.
  • Since React 16.8 (Hooks), function components can do everything class components can — the React team recommends function components for all new code; classes are legacy but still supported.
// Class
class Counter extends React.Component {
  state = { count: 0 };
  render() {
    return <button onClick={() => this.setState({ count: this.state.count + 1 })}>
      {this.state.count}
    </button>;
  }
}

// Function + Hooks
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
Interview tip: If asked which to use today: function components + Hooks, always — classes are only seen in legacy codebases now.
Like you're 5

JSX is like writing a recipe in your own shorthand notes. The kitchen (the browser) doesn't understand your shorthand — a translator (Babel) rewrites it into plain instructions first.

  • JSX (JavaScript XML) is a syntax extension that lets you write HTML-like markup directly inside JavaScript.
  • Browsers cannot execute JSX directly — it isn't valid JavaScript.
  • A compiler/transpiler (usually Babel, or the bundler's built-in transform in tools like Vite) converts JSX into plain React.createElement() (or the newer automatic JSX runtime's jsx()) calls before the code ever reaches the browser.
  • JSX is optional — you could write raw createElement calls — but JSX is far more readable and is the de-facto standard.
// You write:
const el = <h1 className="title">Hi</h1>;

// Babel compiles it to:
const el2 = React.createElement('h1', { className: 'title' }, 'Hi');
Interview tip: Good follow-up point: since React 17's new JSX transform, you no longer need to `import React` in every file just to use JSX.
Like you're 5

Instead of every browser speaking its own dialect of "click," React hires one translator (SyntheticEvent) who speaks the same language everywhere, so your code doesn't have to worry which browser it's running in.

  • React wraps native browser events in a cross-browser wrapper called a SyntheticEvent, giving a consistent API (e.target, e.preventDefault(), etc.) across all browsers.
  • You attach handlers using camelCase JSX props like onClick, onChange, onSubmit — passing a function reference, not a string.
  • Historically React attached one listener at the document root and used event delegation for performance; since React 17 it delegates to the root DOM container the app is mounted in instead of document.
  • Synthetic events are pooled/reused in older React versions for performance (this was removed in React 17+, so accessing the event asynchronously is now safe without calling e.persist()).
function Form() {
  function handleSubmit(e) {
    e.preventDefault();
    console.log('submitted');
  }
  return <form onSubmit={handleSubmit}><button>Send</button></form>;
}
Interview tip: Mention event delegation and that handlers are attached to the root container, not each individual DOM node — that's a performance detail interviewers like to probe.

A Fragment is like an invisible tray that lets you carry several plates to the table without needing an extra visible tray underneath them.

  • A component's return must produce a single root node. Wrapping multiple sibling elements in an extra <div> just to satisfy that rule adds unnecessary markup to the DOM.
  • <React.Fragment> (or the shorthand <></>) groups a list of children without adding an extra DOM node.
  • This avoids breaking CSS (e.g. flex/grid layouts that depend on direct children) and keeps the DOM tree cleaner and slightly lighter.
  • The full <React.Fragment key={id}> form is needed (not the shorthand) when you need to pass a key, e.g. when returning fragments from a list .map().
function Row() {
  return (
    <>
      <td>Name</td>
      <td>Age</td>
    </>
  );
}
Interview tip: Bring up the layout-breaking scenario — e.g. a table expecting direct children breaks if you wrap cells in a stray
.
Like you're 5

A controlled input is a puppet where React holds every string. An uncontrolled input is a puppet that moves on its own, and you only glance at it when you need to check its position.

  • Controlled component: the form element's value is driven entirely by React state (value + onChange). React is the single source of truth.
  • Uncontrolled component: the DOM itself holds the value; React reads it only when needed via a ref (e.g. on submit), using defaultValue instead of value.
  • Controlled inputs give you instant validation, conditional disabling, and formatting as the user types, at the cost of a re-render per keystroke.
  • Uncontrolled inputs are simpler and slightly more performant for very large/simple forms, and are handy when integrating non-React code or file inputs (which can't be controlled).
// Controlled
function ControlledInput() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

// Uncontrolled
function UncontrolledInput() {
  const inputRef = useRef(null);
  const handleSubmit = () => console.log(inputRef.current.value);
  return <input ref={inputRef} defaultValue="" />;
}
Interview tip: File inputs () are always uncontrolled in React because the browser doesn't allow setting their value programmatically.
Like you're 5

If two kids keep arguing over one toy, you don't give each kid their own copy — you give the toy to their shared parent, who hands it to whichever kid needs it.

  • When two or more sibling components need to share and stay in sync with the same piece of state, you move ("lift") that state up to their closest common ancestor.
  • The ancestor holds the state and passes it down as props, along with callback functions the children call to request changes.
  • This keeps a single source of truth instead of duplicated, potentially out-of-sync copies of the same data in multiple components.
  • It's the foundational pattern behind React's unidirectional data flow, and is often the first stepping stone toward reaching for Context or a state library when lifting gets too deep.
function Parent() {
  const [temp, setTemp] = useState(20);
  return (
    <>
      <Slider temp={temp} onChange={setTemp} />
      <Display temp={temp} />
    </>
  );
}
Interview tip: If lifting state pushes it up many levels and causes prop drilling, that's your cue to mention Context API or a state manager as the next step.
Like you're 5

Think of a plant: it gets planted (mount), it grows and you water it whenever something changes (update), and eventually it's removed from the pot (unmount). Hooks let you hook into each of these moments.

  • Function components don't have named lifecycle methods; instead Hooks, mainly useEffect, model the same three phases.
  • Mount (like componentDidMount): useEffect(() => {...}, []) — empty dependency array, runs once after the first render.
  • Update (like componentDidUpdate): useEffect(() => {...}, [dep]) — runs whenever a listed dependency changes.
  • Unmount (like componentWillUnmount): the function returned from inside useEffect is the cleanup, called before the component is removed (or before the effect re-runs).
  • Mentally it's better to think of effects as "synchronizing with an external system" rather than mapping 1:1 onto old lifecycle names.
useEffect(() => {
  console.log('mounted / dep changed');
  const id = setInterval(tick, 1000);

  return () => {
    console.log('cleanup before unmount / before next run');
    clearInterval(id);
  };
}, [dep]);
Interview tip: React 18 Strict Mode intentionally mounts, unmounts, and remounts components in development to surface effects that don't clean up properly — good detail to mention.
Like you're 5

prop-types is like a note on a mailbox that says "packages only, please" — but it's only checked when the mail carrier delivers (at runtime). TypeScript is like a security guard checking every package before it's even shipped.

  • prop-types is a library for runtime type-checking of a component's props — it warns in the console during development if a prop has the wrong type.
  • It doesn't catch errors until the code actually runs, doesn't help with editor autocomplete, and adds a small runtime cost.
  • TypeScript checks types at compile time, before the code ever runs, catching many bugs earlier and enabling rich IDE autocomplete/refactoring.
  • TypeScript also type-checks the rest of your app (not just props) — hooks, function returns, API responses — which is why most modern React codebases have moved to TypeScript and dropped prop-types entirely.
// prop-types (runtime, JS only)
Greeting.propTypes = { name: PropTypes.string.isRequired };

// TypeScript (compile-time)
type Props = { name: string };
function Greeting({ name }: Props) { return <h1>Hi {name}</h1>; }
Interview tip: Framing: prop-types = seatbelt you notice after the crash; TypeScript = the crash never happens because the car wouldn't start.
Like you're 5

Prop drilling is like passing a note through five people in a line just to reach the sixth person — most of those five don't even care what the note says, they're just passing it along.

  • Prop drilling is passing data through many intermediate components that don't use it themselves, just to get it to a deeply nested child.
  • Context API (createContext + useContext) lets a deeply nested component read a value directly without every component in between declaring it as a prop.
  • Component composition — passing components as children instead of data — often avoids the need for deep drilling in the first place.
  • For larger apps with lots of shared, frequently-changing state, a dedicated state management library (Zustand, Redux, Jotai) is often cleaner than Context, since Context re-renders every consumer on any value change unless carefully split.
const ThemeContext = createContext('light');

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Toolbar />
    </ThemeContext.Provider>
  );
}

function DeeplyNestedButton() {
  const theme = useContext(ThemeContext); // no props passed through Toolbar!
  return <button className={theme}>Click</button>;
}
Interview tip: Good nuance: Context solves prop drilling, not necessarily performance — mention splitting contexts or memoizing consumers to avoid over-rendering.
Like you're 5

An HOC is like gift-wrapping a toy in a fancier box before handing it over. Render Props is like handing someone the toy's instructions and letting them decide exactly how to display it.

  • Both are patterns for sharing reusable logic between components, from before Hooks existed.
  • HOC: a function that takes a component and returns a new, enhanced component — const Enhanced = withData(MyComponent). Can cause "wrapper hell" (many nested wrapper components) and prop-name collisions.
  • Render Props: a component takes a function as a prop (often called render or passed as children) and calls it with internal data, letting the consumer decide what to render.
  • Both patterns are largely superseded by custom Hooks today, which share logic without extra component wrapping or nesting at all.
// HOC
function withMousePosition(Component) {
  return function Wrapped(props) {
    const pos = useMousePosition();
    return <Component {...props} pos={pos} />;
  };
}

// Render Props
function MouseTracker({ render }) {
  const pos = useMousePosition();
  return render(pos);
}
<MouseTracker render={pos => <p>{pos.x},{pos.y}</p>} />

// Modern equivalent: custom Hook
function useMousePosition() { /* ... */ }
Interview tip: Strong answer: "Both are legacy patterns for logic reuse; today a custom Hook does the same job with less nesting and clearer code."
Like you're 5

Global styles are like painting the whole house one color — everyone in every room sees it. Modular/scoped CSS is like giving each room its own paint that can't leak into the hallway.

  • Global CSS (a plain .css file imported once, e.g. in the entry point) applies app-wide — good for resets, typography defaults, CSS variables/design tokens.
  • CSS Modules (Component.module.css) auto-generate unique class names per file at build time, scoping styles to just that component and avoiding collisions.
  • CSS-in-JS (styled-components, Emotion) writes styles in JS, colocated with the component, with dynamic props-based styling — at some runtime/bundle-size cost.
  • Utility-first CSS (Tailwind) applies small, composable utility classes directly in JSX, avoiding custom CSS files almost entirely.
  • Modern trend: Tailwind or CSS Modules for most teams, since both avoid CSS-in-JS's runtime overhead while still keeping styles scoped/predictable.
/* Button.module.css */
.primary { background: blue; }

// Button.jsx
import styles from './Button.module.css';
function Button() { return <button className={styles.primary}>Go</button>; }
// compiled className is unique, e.g. "Button_primary__a3F9k"
Interview tip: Mention CSS specificity/collision problems as the core reason scoping solutions (Modules, CSS-in-JS, Tailwind) exist at all.
Like you're 5

StrictMode is like a strict practice coach who makes you repeat a drill twice, on purpose, just to make sure you're not secretly cutting corners.

  • <React.StrictMode> is a development-only wrapper that helps surface potential problems early — it adds zero extra behavior or DOM in production.
  • In development, it intentionally double-invokes certain functions (component function bodies, and since React 18, also mount → unmount → mount) to help detect side effects that aren't properly cleaned up.
  • It warns about legacy/deprecated APIs (old lifecycle methods, legacy context, findDOMNode), and helps you find state updates or effects that aren't idempotent/pure.
  • It does not catch every bug and has no effect on the production build's performance or behavior — it's purely a development-time safety net.
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);
Interview tip: If someone complains "my useEffect runs twice!", explain it's StrictMode in dev mode doing double-invoke on purpose to catch missing cleanup — not a bug, and won't happen in production.
Like you're 5

Sometimes the simplest way to hide something is to just... not put it on the table at all. Returning null is React's way of saying "nothing here."

  • Return null (or undefined/false) from a component's function — React renders nothing for that component, though it's still "mounted" in the tree.
  • Use conditional rendering in the parent so the component isn't even referenced when not needed.
  • React.memo() prevents unnecessary re-renders (not the initial render) when props haven't changed — different from preventing rendering entirely.
  • In class components, shouldComponentUpdate() returning false skips a re-render — the function-component equivalent is React.memo with a custom comparison function.
function WarningBanner({ show, message }) {
  if (!show) return null; // renders nothing
  return <div className="warn">{message}</div>;
}
Interview tip: Distinguish clearly: returning null = "don't render this output"; React.memo = "don't re-render if props are unchanged." Interviewers often test if you conflate the two.
Like you're 5

A picky chef only recooks a dish if the ingredients actually changed. If you hand back the exact same ingredients, the chef says "nothing's different, here's the same plate as before" instead of cooking again from scratch.

  • React.memo(Component) wraps a function component so React skips re-rendering it if its props are shallowly equal to the previous render's props.
  • The class equivalent is React.PureComponent, which implements a shallow-compare shouldComponentUpdate automatically.
  • "Shallow" comparison only checks one level deep (=== per prop) — new object/array/function props created fresh on every render (e.g. onClick={() => ...} inline) will still be seen as "different" and defeat memoization.
  • That's why React.memo is usually paired with useMemo (for object/array props) and useCallback (for function props) in the parent, so the same reference is reused across renders.
  • Only worth adding where a component is expensive to render or re-renders very frequently — it adds its own comparison overhead, so it's not a free performance win everywhere.
const ExpensiveRow = React.memo(function Row({ item }) {
  console.log('rendering', item.id);
  return <li>{item.label}</li>;
});

// Parent must also keep props stable:
const item = useMemo(() => ({ id, label }), [id, label]);
Interview tip: Classic gotcha to mention: wrapping a component in React.memo does nothing if its parent keeps passing brand-new inline object/function props each render.
