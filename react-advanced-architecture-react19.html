Imagine a chef preparing a dish entirely in the kitchen and just bringing you the finished plate — you never see the raw ingredients or the cooking mess, and your table stays clean and light.

  • Server Components are components that render only on the server — their code (and its dependencies) never gets sent to the browser as JavaScript.
  • They can directly access server-only resources — databases, the filesystem, secrets — without an API layer in between, and their output (already-rendered UI description) is streamed to the client.
  • Because their JS never ships to the browser, they don't add to the client bundle size, which is a major win for apps with heavy dependencies (e.g. a markdown parser) used only for rendering.
  • They can't use browser-only features (state, effects, event handlers, window) — anything interactive must be a Client Component, marked with 'use client', and the two compose together in the same tree.
Interview tip: Be precise: RSC is a rendering model, not a replacement for SSR — many apps mix Server Components, Client Components, and traditional SSR together.
Like you're 5

Traditional SSR is like mailing someone a photo of a finished puzzle, but they still have to rebuild the entire puzzle themselves at home. Server Components mail the finished puzzle itself — no rebuilding required for the parts that don't move.

  • Traditional SSR renders the entire component tree to HTML on the server for the first paint, but then ships the full JavaScript for every component so the client can hydrate (re-run all component code, attach event listeners) — even for parts of the page that never change.
  • Server Components render on the server and never re-render or hydrate on the client at all — their JS is simply never sent, since there's nothing for the client to do with them.
  • SSR is about when rendering happens (server, for the first paint); RSC is about where a component's code is allowed to live at all (server-only, forever) — they're complementary, not competing, concepts.
  • A page can mix both: Server Components for static/data-heavy sections, and Client Components (still potentially SSR'd for the first paint, then hydrated) for interactive widgets.
Interview tip: This is a very commonly confused pair — nailing the distinction between "when" (SSR) and "where forever" (RSC) is what separates a strong answer from a shaky one.

It's like a big gate at the edge of a garden. Everything before the gate is peaceful and hidden from visitors (server-only). The moment you cross the gate, everyone can see and touch it (client-side, interactive).

  • 'use client' is a directive placed at the top of a file, marking that component (and everything it imports) as a Client Component — it and its subtree will be bundled and shipped to, and can run in, the browser.
  • It defines the boundary where server-only code stops and browser-capable, interactive code begins; everything imported from a 'use client' file is also considered part of the client bundle.
  • By default (without the directive), components are treated as Server Components — 'use client' is an opt-in escape hatch for interactivity, not the default mode.
  • Once you cross into a Client Component, you can use hooks, state, effects, and event handlers freely — but you can no longer directly import and render a Server Component inside it (you'd pass it down as a prop/children from a parent instead).
'use client';
import { useState } from 'react';

export default function LikeButton() {
  const [liked, setLiked] = useState(false);
  return <button onClick={() => setLiked(!liked)}>{liked ? '❤️' : '🤍'}</button>;
}
Interview tip: Mention that the boundary is per-file and "infectious" downward through imports — a common gotcha in real projects.

You can't hand a live, wriggling puppet across a border where puppets aren't allowed — unless it's a special certified "Server Action" puppet that's built to cross safely.

  • Generally no — regular JavaScript functions can't be serialized across the server/client network boundary, so passing an arbitrary function prop from a Server Component down into a Client Component will error.
  • The one exception: Server Actions (functions marked 'use server') are specially serializable — React/the framework wires up a reference so the client can call back into that specific server function (e.g. as a form's action).
  • The typical pattern instead: pass data (already-rendered results, plain serializable values) from Server to Client Components, and keep event-handling logic (onClick, onChange) defined locally within Client Components themselves.
  • You can pass a Server Component itself as children/props into a Client Component (rendered content, not a function) — that's a valid and common composition pattern.
Interview tip: Distinguish clearly: passing a rendered Server Component as children into a Client Component is fine; passing an arbitrary plain function is not, unless it's a Server Action.

Instead of waiting for every single course of a meal to be ready before serving anything, the waiter brings out the salad the moment it's ready, then the soup, then the main course — you start eating right away instead of staring at an empty table.

  • Streaming lets the server send HTML to the browser in chunks as it becomes ready, instead of waiting for the entire page to finish rendering before sending anything.
  • <Suspense fallback={}> marks a boundary: React shows the fallback immediately for that section while the wrapped component's data is loading, then streams in the real content and swaps it in once ready — without blocking the rest of the page.
  • This lets fast parts of a page (navigation, header) appear instantly, while slow parts (a data-heavy widget) load in later, each independently, dramatically improving perceived performance.
  • Multiple Suspense boundaries can be nested, letting different sections of the same page resolve and stream in independently, in whatever order their data becomes ready.
<Suspense fallback={<Spinner />}>
  <SlowUserProfile userId={id} /> {/* streams in once its data resolves */}
</Suspense>
Interview tip: Frame it as solving the "all-or-nothing" waterfall problem of traditional SSR — that's the actual value proposition interviewers are checking for.

Instead of setting up a whole waiting room with a receptionist just to check if your package arrived, use lets you just peek directly and grab it the moment it's there — right in the middle of what you're doing.

  • use(promise) lets you read the value of a Promise directly during render — if the Promise hasn't resolved yet, React suspends that component (works together with a <Suspense> boundary above it) until it does.
  • use(context) can also read a Context value, similarly to useContext, but — unlike other Hooks — use can be called conditionally (inside an if), since it isn't bound by the usual Rules of Hooks call-order requirement.
  • It's commonly used to consume a Promise passed down from a Server Component (data fetched on the server) directly inside a Client Component, integrating naturally with streaming and Suspense.
  • Different from a normal Hook: it's designed specifically to be usable in loops, conditionals, and early returns — making certain conditional-data-reading patterns cleaner than they were before.
function Profile({ userPromise }) {
  const user = use(userPromise); // suspends until resolved
  return <h1>{user.name}</h1>;
}

<Suspense fallback={<Spinner />}>
  <Profile userPromise={fetchUser(id)} />
</Suspense>
Interview tip: The detail that use() can be called conditionally, unlike other Hooks, is a favorite "gotcha" follow-up question — know it by heart.

A search engine is like a visitor peeking through your window from outside before deciding whether to come in — if the important sign in your window (title, description) only appears after they've already walked away, they never see it.

  • Search engine crawlers and social-media link previews mostly need <title>, <meta name="description">, and Open Graph tags present in the initial HTML response, not injected later by client-side JavaScript after hydration.
  • In meta-frameworks (Next.js App Router, Remix), you export a generateMetadata()/meta() function per route that can be async and fetch the specific data needed (e.g. a blog post's title) to build accurate, per-page metadata server-side.
  • For a plain client-rendered SPA without SSR, dynamic per-page metadata is much harder for crawlers to see reliably — this is one of the strongest practical reasons to reach for SSR/RSC for any content-focused, SEO-sensitive site.
  • Structured data (JSON-LD <script type="application/ld+json">) can be injected the same server-rendered way to enable rich search results (ratings, breadcrumbs, etc).
// Next.js App Router example
export async function generateMetadata({ params }) {
  const post = await getPost(params.slug);
  return { title: post.title, description: post.summary };
}
Interview tip: If the app is a pure client-rendered SPA, be honest that dynamic SEO there is a real limitation — pushing back on an unrealistic SEO requirement for a pure SPA is a legitimate, senior-level answer.

The server sends a photo of a finished cake. Hydration is the client baking that exact same cake itself and then carefully swapping in the real one behind the photo. A mismatch is when the client's cake looks slightly different from the photo — confusing everyone.

  • Hydration is the process where React, on the client, re-renders the component tree over the server-rendered HTML and attaches event listeners and internal state to the existing DOM nodes, rather than re-creating them from scratch.
  • A hydration mismatch occurs when the HTML React renders on the client doesn't match what the server actually sent — React detects the difference and either warns and patches it (slower, can cause a visible flicker) or, in some cases, discards and re-renders the whole subtree.
  • Common causes: using browser-only values during render (window, localStorage, Date.now(), Math.random() for IDs), rendering different content based on user-agent detection, or invalid HTML nesting that browsers silently "fix" differently than React expects.
  • Fix: keep the very first render identical between server and client — use a default/placeholder value for anything browser-only, and update it only after mount inside a useEffect.
// ❌ Mismatch risk
function Clock() { return <p>{new Date().toLocaleTimeString()}</p>; }

// ✅ Safe: same on server and client's first render
function Clock() {
  const [time, setTime] = useState(null);
  useEffect(() => setTime(new Date().toLocaleTimeString()), []);
  return <p>{time ?? '--:--:--'}</p>;
}
Interview tip: Cite Math.random()/Date.now()/window usage during render as the concrete, memorable cause list — interviewers love a specific example over an abstract definition.

A circuit breaker in your house stops one faulty appliance from burning down the whole building — it isolates the damage to just that one room instead of taking out all the lights everywhere.

  • An Error Boundary is a component that catches JavaScript errors thrown during rendering, in lifecycle methods, or in constructors of its child tree, and displays a fallback UI instead of crashing the entire app.
  • Error Boundaries must currently be written as class components implementing static getDerivedStateFromError() and/or componentDidCatch() — there is no Hook equivalent yet, which is exactly why libraries exist to wrap this boilerplate.
  • They do not catch errors in event handlers, async code, server-rendering, or errors thrown in the boundary itself — only errors thrown during the render phase of descendants.
  • react-error-boundary provides a ready-made <ErrorBoundary FallbackComponent={...} onReset={...}> component and a useErrorBoundary hook to manually trigger the boundary from event handlers/async code, avoiding writing the class boilerplate yourself.
import { ErrorBoundary } from 'react-error-boundary';

<ErrorBoundary FallbackComponent={ErrorFallback} onReset={() => refetch()}>
  <UserProfile />
</ErrorBoundary>
Interview tip: Always mention what Error Boundaries DON'T catch (event handlers, async code) — it's the single most common follow-up trap in this question.

A portal is like a magic door: your character stays logically part of the same story (same React tree), but visually they've stepped into a completely different room (a different spot in the actual page).

  • createPortal(children, domNode) renders a component's children into a different DOM node than where the component logically sits in the React tree — while still remaining part of the same React component tree (context, event bubbling, and state all work normally).
  • Classic use case: modals, tooltips, dropdowns — you want them rendered at the very top of the actual DOM (often directly under <body>) so they aren't clipped by a parent's overflow: hidden or constrained by z-index stacking contexts, but you still want them to behave like any other React component (receive props, use context, etc).
  • Events triggered inside a portal still bubble up through the React tree as if it weren't a portal at all — e.g. a click inside a modal still triggers a parent's onClick handler in React's synthetic event system, even though it's a different DOM subtree entirely.
  • The target DOM node (e.g. document.getElementById('modal-root')) must already exist in the HTML, typically added as an extra empty <div> alongside the app's root element.
function Modal({ children }) {
  return createPortal(
    <div className="modal-overlay">{children}</div>,
    document.getElementById('modal-root')
  );
}
Interview tip: Mention explicitly that React events still bubble through the LOGICAL tree, not the DOM tree — that's the detail that trips people up and impresses when explained correctly.

You wouldn't pack your entire closet for a weekend trip — you bring what you need now, and can always grab the rest later if it turns out you need it.

  • Code splitting breaks a JavaScript bundle into smaller chunks that are loaded on demand, rather than shipping one giant bundle upfront — reducing the initial load time for users.
  • React.lazy(() => import('./Component')) defines a component whose code is fetched only when it's first rendered, typically wrapped in a <Suspense fallback={...}> to show a loading state while the chunk downloads.
  • Commonly applied at route boundaries (each page/route is its own chunk) so users only download the code for the page they're actually visiting.
  • Bundlers (Webpack, Vite) automatically create a separate chunk file for each dynamic import() call — React.lazy is the React-level API that ties that mechanism into rendering and Suspense.
const Settings = React.lazy(() => import('./Settings'));

<Suspense fallback={<Spinner />}>
  <Settings />
</Suspense>
Interview tip: Mention route-level splitting as the most common, highest-value place to apply this — a blanket "lazy load everything" answer misses the real trade-off (more requests, more loading states).

Instead of a blank grey box while a photo loads, you show a soft, blurry outline shaped like the photo that's coming — so it feels like it's already almost there.

  • Wrap the section that depends on async data in <Suspense fallback={}> — the fallback is shown immediately while the real content is loading/suspended.
  • The skeleton component should mimic the shape and layout of the eventual real content (same approximate width/height, similar element structure) so there's no jarring layout shift once the real content swaps in.
  • Multiple independent Suspense boundaries let different parts of a page show their own skeletons and resolve independently — e.g. a page header appears instantly while a slow comments section still shows its own skeleton.
  • Combine with useTransition when switching between already-loaded views, so a brief pending state (not necessarily a full skeleton) can be shown instead of a jarring flash back to a fallback.
<Suspense fallback={<ProfileSkeleton />}>
  <ProfileCard userId={id} />
</Suspense>
Interview tip: Emphasize matching the skeleton's dimensions to the real content to avoid layout shift — that's a concrete performance/UX metric (Cumulative Layout Shift) interviewers may probe.

Imagine having an assistant who automatically notices which parts of your homework didn't change since last time and quietly skips redoing just those parts — without you having to circle and label them yourself.

  • The React Compiler (previously called React Forget) is a build-time tool that automatically analyzes your component code and inserts memoization — effectively auto-generating the equivalent of useMemo, useCallback, and React.memo where beneficial.
  • It relies on your components following the Rules of React (pure render functions, no illegal side effects during render) to safely reason about what's actually safe to skip recomputing.
  • In practice, this means far less manual memoization boilerplate is needed going forward, and fewer subtle bugs from incomplete/incorrect dependency arrays written by hand.
  • It doesn't remove the Hooks themselves from the language — you can still use useMemo manually for cases the compiler can't infer, or in codebases not yet using the compiler.
Interview tip: Position this as "manual memoization becomes the exception, not the default" going forward — a good forward-looking answer for a fast-moving ecosystem question.

Instead of one giant Lego castle built and owned by a single team, several teams each build their own separate section and they all snap together into one big castle at the end.

  • Micro-frontends split a large application into independently built, tested, and deployed pieces — often owned by separate teams — that are composed together at runtime (or build time) into a single user-facing app.
  • Module Federation (a Webpack/Rspack feature) is the most common modern approach: each micro-frontend exposes specific modules/components that a "host" app can dynamically load and mount at runtime, without a full rebuild of the host.
  • Alternative approaches: iframes (strong isolation, but poor UX/communication), or build-time composition (each team publishes a package, one build assembles them all — simpler, but loses independent deployability).
  • Key challenges to call out: shared dependency versions (avoiding shipping multiple React copies), consistent design system/styling across teams, and cross-app communication (often via a shared event bus or URL-based routing).
Interview tip: Naming Module Federation specifically, plus at least one real trade-off (shared dependency versioning), signals hands-on architecture experience rather than textbook knowledge.

If you order a box of assorted tools but only ever use the hammer, tree shaking is like the delivery truck noticing that and only packing the hammer, leaving the rest back at the warehouse.

  • Tree shaking is a build-time optimization where the bundler analyzes which exports from a module are actually imported and used, and removes ("shakes off") the rest from the final bundle.
  • It relies on ES Modules' static import/export syntax, since the bundler can analyze imports without running the code — CommonJS's dynamic require() is much harder to tree-shake reliably.
  • For it to work well, both your own code and the libraries you import need to ship ES Modules (often marked with a "sideEffects": false field in package.json) — a library shipped only as CommonJS, or with unclear side effects, may pull in far more code than you'd expect.
  • Directly affects final bundle size (and therefore load time) — a common real-world example is importing one function from a large utility library carelessly (import _ from 'lodash') versus a shakeable, named import (import { debounce } from 'lodash-es').
// Poor tree-shaking (imports the whole library)
import _ from 'lodash';
_.debounce(fn, 300);

// Tree-shakeable (only the used function is bundled)
import { debounce } from 'lodash-es';
debounce(fn, 300);
Interview tip: The lodash vs lodash-es example is a genuinely common real-world case — using it shows practical bundle-size awareness, not just theory.

A TV remote's buttons don't work alone floating in space — they only make sense together, sharing one hidden connection to the same TV. Compound components are pieces that only make full sense used together, quietly sharing state behind the scenes.

  • The Compound Components pattern lets several related components work together to manage shared state, while giving the consumer full control over layout — e.g. <Select>, <Select.Option>, <Select.Trigger>.
  • The parent (<Select>) holds the shared state and exposes it to its children via Context, so each child (<Select.Option>) can read/update it without the consumer having to manually wire up props between siblings.
  • This gives a very flexible, composable API — the consumer can reorder, conditionally omit, or wrap sub-components freely, unlike a single monolithic component with a long list of configuration props.
  • Common in component libraries (Radix UI, Headless UI, many design systems) precisely because it separates behavior/state (owned by the pattern) from markup/styling (owned by the consumer).
const SelectContext = createContext();

function Select({ children, value, onChange }) {
  return <SelectContext.Provider value={{ value, onChange }}>{children}</SelectContext.Provider>;
}
Select.Option = function Option({ value, children }) {
  const ctx = useContext(SelectContext);
  return <div onClick={() => ctx.onChange(value)}>{children}</div>;
};

// Usage: <Select value={v} onChange={setV}><Select.Option value="a">A</Select.Option></Select>
Interview tip: Naming a real library that uses this pattern (Radix UI) grounds the answer in something interviewers likely recognize themselves.

Normally a toy manages its own on/off switch by itself. The Control Props pattern lets a parent optionally grab the switch and flip it themselves instead, while the toy still works fine on its own if nobody bothers to take over.

  • The Control Props pattern lets a component be either uncontrolled (it manages its own internal state, e.g. an accordion tracking its own open/closed state) or controlled (the parent passes a value + onChange and fully owns the state) — using the same component API for both.
  • Implementation checks whether a given prop (e.g. value) was provided: if it was, the component uses that value and calls the onChange callback instead of updating its own internal state; if not, it falls back to managing its own internal state.
  • This mirrors exactly how native HTML form elements like <input> behave (controlled with value+onChange, or uncontrolled with defaultValue) — it's a very familiar mental model to build custom components around.
  • Gives library authors flexibility: simple use-cases work out of the box uncontrolled, but advanced consumers who need to synchronize state elsewhere (URL params, another component) can take full control without needing a different component.
function useControllableState(controlledValue, defaultValue, onChange) {
  const [internal, setInternal] = useState(defaultValue);
  const isControlled = controlledValue !== undefined;
  const value = isControlled ? controlledValue : internal;
  const setValue = (next) => {
    if (!isControlled) setInternal(next);
    onChange?.(next);
  };
  return [value, setValue];
}
Interview tip: Tie it back to native 's value/defaultValue duality — that comparison instantly clicks for most interviewers.

It's a component that hands you all the useful ingredients and instructions, but leaves it completely up to you to actually plate and decorate the final dish however you like.

  • A renderless component encapsulates behavior/state/logic but renders no markup of its own — it exposes data and functions (via a render prop, or these days more commonly a custom Hook) and leaves 100% of the visual output to the consumer.
  • Classic pre-Hooks example: a <Toggle> component that takes a render or children function prop, calls it with { isOn, toggle }, and renders exactly whatever that function returns.
  • The pattern's core goal — separating reusable logic from specific presentation — is now most commonly achieved with a custom Hook instead, since a Hook accomplishes the same separation without any extra component/render-prop wrapping in the tree at all.
  • Still occasionally useful when the logic genuinely needs to inject event handlers into deeply specific markup structure that a Hook alone can't cleanly express, but it's increasingly rare in modern codebases.
// Renderless (render-prop) version
function Toggle({ children }) {
  const [isOn, setIsOn] = useState(false);
  return children({ isOn, toggle: () => setIsOn(v => !v) });
}

// Modern equivalent: a custom Hook
function useToggle() {
  const [isOn, setIsOn] = useState(false);
  return { isOn, toggle: () => setIsOn(v => !v) };
}
Interview tip: Be ready to say plainly that custom Hooks have mostly replaced this pattern in modern code — a common trap is describing it as if it's still the state of the art.

A design system is like a shared box of LEGO pieces that always look consistent no matter who builds with them — every team reaches for the same blue brick instead of everyone painting their own slightly-different shade.

  • Define design tokens first — colors, spacing scale, typography, radii — as a single source of truth (CSS variables, a Tailwind config, or a JS theme object), so every component pulls from the same values instead of hardcoding one-off styles.
  • Build a layer of low-level primitive components (Button, Input, Stack, Card) that consume those tokens, then compose higher-level, feature-specific components out of the primitives rather than duplicating styling logic everywhere.
  • With Tailwind, tokens live in tailwind.config.js and components use utility classes referencing them; with Styled Components/Emotion, tokens typically live in a ThemeProvider object accessed via a styling function.
  • Document and enforce the system with a tool like Storybook (isolated component playground/visual testing) so consumers discover what already exists before building a one-off variant, which is usually how design systems drift and fragment over time.
Interview tip: Mentioning Storybook (or an equivalent) as the governance/discovery layer is what separates "I can style a button" from "I've actually maintained a design system."

A Shadow DOM is like a soundproof, private room inside a house — the noise (styles/markup) inside stays contained and doesn't leak out to bother the rest of the house, and outside noise doesn't leak in either.

  • Shadow DOM is a native browser API that attaches a hidden, encapsulated DOM subtree to an element — styles defined inside it don't leak out, and outside CSS doesn't leak in, giving true style/markup isolation.
  • React does not use Shadow DOM by default — React's component encapsulation is purely a JavaScript-level abstraction (component boundaries, props), rendered into the regular, shared DOM tree where global CSS can still affect it.
  • React can render inside a Shadow DOM root manually (e.g. when building a Web Component that needs true style isolation, such as an embeddable third-party widget), but this requires extra setup, since React's synthetic event system and some APIs historically assumed a single, non-shadow document root.
  • This is exactly why teams reach for CSS Modules, CSS-in-JS, or Tailwind's utility classes for scoping in typical React apps — they solve the same "style collision" problem Shadow DOM solves natively, but at the tooling/convention level instead of the browser level.
Interview tip: The key soundbite: React solves style scoping via tooling/convention (CSS Modules, CSS-in-JS); the browser solves it natively via Shadow DOM — React doesn't use the latter by default.
